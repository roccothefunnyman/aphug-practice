# CLAUDE.md

Context for Claude Code when working in this repo.

## What this project is

A static quiz app for **AP Human Geography** practice. **120 multiple-choice questions across 2 sets**, generated by Claude on claude.ai for Rocco's daughter to use studying for the AP exam. The app is plain HTML / CSS / vanilla JS.

The repo is published as a public GitHub Pages site so she can study from any device.

120 questions total across two independent sets (60 each), all multiple-choice (A-D). No case studies. No FRQs.

Set 1 ids are `q01..q60` (no prefix, preserved so existing localStorage mastery survives). Set 2 ids are `s2-q01..s2-q60`. Each question record carries a `set` field (1 or 2) and a `set_label`.

Each set is a complete 60-question practice test mirroring the official AP HuG MC distribution. New sets can be appended by adding entries to `SETS` in `build.py`.

## AP HuG units

The College Board CED groups AP Human Geography into 7 units. Per-unit progress bars and picker chips are driven by these:

1. Thinking Geographically
2. Population & Migration
3. Cultural Patterns & Processes
4. Political Patterns & Processes
5. Agriculture & Rural Land Use
6. Cities & Urban Land Use
7. Industrial & Economic Development

Current bank distribution (`py build.py` prints this on every build): U1=15, U2=6, U3=7, U4=8, U5=10, U6=5, U7=9.

## Source file format

Questions and answers live in single markdown files per set (not per-question files like the AB-100 sibling project). The build script handles **two slightly different markdown formats** since each set of questions Claude generated arrived with different conventions.

**Format A (Set 1):**
- Questions: split by `## Unit N: <Name>` and `### Question N` headers; options as `A) ...`.
- Answers: `**QN: <letter> — <headline>**` then 1-3 explanation paragraphs, then a footer line `*Topic X.Y | Skill N.A*`.

**Format B (Set 2):**
- Questions: split by `## Unit N: <Name> (Q...)` headers and inline `**QN.**` headers (stem starts on same line); options as `A. ...`.
- Answers: `**QN. <letter>.** <explanation, ending with inline> Topic X.Y. Skill N.A.` There is no headline; `build.py` synthesizes one from the first sentence of the explanation so the green callout still reads well.

`build.py` detects both formats per file. If you generate a third set in either shape, just add it to `SETS` in `build.py`; no parser changes needed.

Question stems may include markdown tables for data stimuli — these render through marked.js inside the question card.

## Build

```
py build.py
```

Walks each entry in `SETS`, parses its question + answer files, writes `app/questions.json`. The build script:
- Splits questions on either `### Question N` or `**QN.**` and tracks the most recent `## Unit N` header to assign each question its unit.
- Strips `*End of practice test.*`, trailing `---` rules, and the `## Scoring Rubric` / `## How to Review` tail.
- Pulls out the option letters and their text (handles `A)` and `A.`).
- Pairs each question with its answer block by question number, within the set.
- Extracts the topic code and skill code from either the `*Topic ... | Skill ...*` footer or the inline `Topic ... Skill ...` trailing sentence.
- Stamps each record with `set`, `set_label`, and an id prefixed by `id_prefix` from the SETS entry.
- Self-contained, no Python dependencies beyond the stdlib.
- Prints the per-set, per-unit count and any unmatched question numbers.

## Run locally

```
cd app
py -m http.server 8000
```

Then http://localhost:8000. Opening `index.html` directly via `file://` will fail because the page uses `fetch()` to load `questions.json`.

## Deploy

GitHub Pages, public repo, branch `main` (or `master`), root folder. URL: `https://<user>.github.io/aphug-practice/app/`.

### Cache busting

`index.html` loads `style.css?v=N`, `voice.js?v=N`, and `app.js?v=N`. **Bump N whenever JS or CSS changes in a way that depends on new markup IDs.** Otherwise browsers may load the new HTML against cached old JS and silently break. Bump in lockstep across all three.

## App features

- **Picker modal.** Topbar `New session` opens it. Quick-pick chips: All / None / Wrong only / Unanswered / Correct only / Set 1 only / Set 2 only / Unit 1-7. Per-unit All/None links. Per-question checkboxes with one-line stem preview, set tag (S1/S2), and mastery dot.
- **Progress bars driven by the active subset.** Top bar `correct / wrong / total` plus per-unit bars below it (7 colored bars, one per AP HuG unit).
- **Mastery persistence.** Per-question correct/wrong stored in `localStorage.aphug_stats_v2`, keyed by question id (`q01`...`q60` for Set 1, `s2-q01`...`s2-q60` for Set 2). `Reset progress` clears it. (The v2 key was introduced when the question content was rewritten to remove a length-bias signal, so old `_v1` mastery was invalidated.)
- **Light / dark theme.** Toggle in the topbar; persisted to `localStorage.aphug_theme`.
- **Voice.** Read-aloud (TTS), push-to-talk spoken answers (ASR), and a voice-only mode that auto-reads each question and listens for an answer. All browser-native via `voice.js`. No API key.
- **Slide controls.** `Back` (`←`), `Clear` (wipe current pick), `Copy Question` (stem + options to clipboard). Multiple-choice auto-reveals on click; no Reveal button needed.
- **Session state.** `sessionStorage.aphug_session_v2` holds `{ order, cursor, salt, version }`. The `salt` drives a deterministic per-question option-letter shuffle so a refresh mid-session keeps the same letter order. Starting a New Session generates a new salt so letters reshuffle between sessions, defeating "the answer is always B" memorization. Closing the tab loses session state. Mastery and theme live in `localStorage` and survive.

- **Per-session option shuffling.** Options A/B/C/D are presented in a permutation derived from `mulberry32(hash(qid + salt))`. The keyed answer letter in `questions.json` is the original markdown letter; the app maps display letter → original letter for grading. Answer-card headline shows the display letter the user actually saw. This was introduced because the question generator concentrated correct answers in the B slot at 65-80% rates.
- **First launch** auto-starts a session with all 120 questions. The picker is only shown when the user clicks `New session` or `Start New Session`. To drill one set, use the `Set 1 only` / `Set 2 only` chip in the picker.

## When adding new questions

**To extend an existing set:**
1. Append the new question block to the right unit section of that set's questions file. Match the existing shape (Format A or B — see "Source file format" above).
2. Add the matching answer block in the corresponding answer file. Include a topic and skill code.
3. Run `py build.py` to refresh `app/questions.json`.
4. Commit and push. If you also touched `app.js`, `voice.js`, or `style.css`, bump the `?v=N` cache-busters in `index.html`.

**To add a new set (Set 3+):**
1. Drop both files (questions + answers) into `source/`.
2. Add a new entry to the `SETS` list in `build.py`: `{"set": 3, "label": "Set 3", "id_prefix": "s3-", "questions": ..., "answers": ...}`.
3. Add a `<button class="chip-btn" data-preset="set-3" type="button">Set 3 only</button>` to the picker presets in `index.html`.
4. Add color rules for `tag-set-3` and `picker-tag.set-tag-3` in `style.css`.
5. Build, bump cache-busters, commit, push.

## Validating new content

These questions were AI-generated, so accuracy is a known concern. Before pushing new questions:
- Spawn parallel sub-agents, ~20 questions each, to validate every stated correct answer against the College Board CED and standard textbooks (Rubenstein, AMSCO, Barron's).
- Each agent should flag wrong keyed answers, ambiguous distractors, factual errors in explanations, and CED-misaligned wording.
- Surface flagged items to Rocco for approval before editing the source files.

The first 60 were validated this way at repo creation; results: 57 fully OK, 3 minor stylistic notes (Q7 distractor ambiguity, Q20 explanation wording, Q48 explanation wording).
